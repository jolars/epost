# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Source of truth: DESIGN.md

`DESIGN.md` is the authoritative architecture document. **Read it before proposing or making non-trivial changes.** It pins the rendering model (TUI via `ratatui` + `crossterm`, with HTML walked through `html5ever` into a Block-IR and drawn as cells; inline images via `ratatui-image`'s kitty/iTerm/sixel/halfblock detection), the storage model (maildir is truth; sqlite is a disposable cache; everything keys on `Message-ID`, never on path), the configuration model (declarative read-only TOML), the concurrency model (`std::thread` + `mpsc` channels, **not** `async`/`await`), and the deployment target (NixOS / Home Manager, fully config-driven). The numbered *Hard requirements / invariants* are real constraints; do not silently relax them. *Suggested build order* is the v1 roadmap.

**Note on history:** this project briefly started as a GUI (eframe + Blitz + wgpu) before pivoting to the TUI design in `DESIGN.md`. The pivot cleanup is done — no GUI code remains. If you see lingering references to wgpu, `NetProvider`, Blitz, eframe, or a "four-panel egui layout" anywhere, that's stale documentation: rip it out, don't build on it.

Auto-`S` fires once per opened body inside `ensure_body_for_selection`; in Normal mode (List focused), `m` toggles Seen, `*` toggles Flagged (`F`, surfaced as a ★ glyph in the list), `x` toggles Trashed (`T`, surfaced as dimmed/strikethrough). Cmdline: `:compose` / `:reply` / `:reply-all` / `:forward` / `:send` / `:close`.
- **Attachments (multipart/mixed)** — done. `Draft.attachments: Vec<PathBuf>` is read at `serialize()` time (not at attach time) so a file deleted between `:attach` and `:send` surfaces as a send-path error rather than silently shipping stale content. Per-file `Content-Type` is derived via `mime_guess::from_path` (falls back to `application/octet-stream`). Cmdline: `:attach <path>` queues a file (tilde-expands `~/`, validates existence + is-file at attach time), `:detach <n>` removes by 1-based index. `ui/compose.rs` renders an `Attach:` row in the header when the list is non-empty.
- **Cross-folder moves** — done. `:archive` / `:spam` / `:trash` route through the per-account `archive_folder` / `spam_folder` / `trash_folder` config keys; `:mv <folder>` takes a literal folder label (Maildir++ encoded — dotless — or fs `/`-joined, per the account's layout). Keys: `a` archives, `d` trashes the selected row (cross-folder move), `D` trashes every member of the row's thread in one shot (one batched undo via `UndoAction::Batch`, per-row routing through each message's own account, rows on accounts with no `trash_folder` are skipped and counted in the status row; disabled in `/` search), `x` toggles the `T` info-flag. The destination path is resolved through `account.layout.folder_path(root, label)`; target `{cur,new,tmp}/` is created if missing. Self-writes are recorded on both source and destination paths for the Step 7 watcher.
- **Step 7 (notify watcher)** — done. `notify`-backed inotify watcher in `store/watch.rs`, per maildir folder (account root non-recursive, plus `{cur,new}` for INBOX and each discovered sub-folder's `{cur,new}`). Folder discovery uses each account's configured `Layout`; under verbatim the watcher additionally treats every folder root as a discovery root so nested sub-folder creation (e.g. `Sent/2024/`) triggers re-discovery — under Maildir++ only the account root is a discovery root since all sub-folders surface as dot-prefixed siblings there. Events route through `SelfWrites::consume` to suppress our own renames, coalesce into per-folder dirty marks, debounce on `[watch].debounce_ms` (default 250 ms), and dispatch through `scan::rescan_folders` — disk I/O restricted to the dirty folders, list + folder stats refreshed from the index. `Index::prune_folder` reflects deletions; `apply_rescan` preserves the selected msgid across rescans (snaps to row 0 when gone). New subfolders auto-register: `:mv NewFolder` calls `Watcher::register_folder` race-free; a Create(Folder) event on any discovery root queues a `Rediscover` that diffs against already-watched folder roots and registers only the new ones. `[watch]` config: `enabled` (default true), `debounce_ms` (default 250). Watcher start failure (e.g. `fs.inotify.max_user_watches` exhausted) degrades to startup-only scan with a one-shot warning. Flag-flip + cross-folder-move callsites switched to `set_flag_recorded` / `move_to_folder_recorded` so `SelfWrites` is populated *before* the rename, closing the race the live watcher would otherwise see.
- **Multi-account UI** — done. Sidebar renders one `[all]` group (unified across accounts, today's default) plus one `[<account>]` group per configured account, alphabetic, each with that scope's folders. `InboxScreen.current_account: Option<String>` (None = `[all]`) plus the existing `current_folder` form the active scope; `Index::list_folder` / `Index::folder_stats` accept `Option<&str>` to filter. Scan workers emit `Vec<AccountFolderStats>`; in-memory stat patching writes both the `[all]` group and the owning account's group. Folder cycling (`Alt-j`/`Alt-k`, `j`/`k` in Folders pane) walks the flat list skipping non-selectable group headers. Top bar carries an `account · folder` badge. `:account <name>` / `:account all` jumps scope from cmdline. Cross-folder moves continue to route per-row via `MessageRow.account`.
- **`:sync` dispatch** — done. `store/sync.rs` spawns the configured `[sync].command` on a `std::thread`, waits for it, and reports the exit status over `mpsc`. `App.sync_rx` holds the in-flight receiver; `App::poll_sync` drains it each tick into the cmdline status row (`syncing…` → `synced` / `sync failed: …`). The worker pushes `AppEvent::Wake` on completion so the result surfaces without waiting for the idle heartbeat. A second `:sync` while one is in flight errors out instead of queueing. The maildir watcher reconciles whatever the sync wrote — this worker only reports the command itself.
- **OSC 8 hyperlinks in the reader** — opt-in via `[reader].osc8_links` (default **on**). `emit_osc8_hyperlinks` in `ui/reader.rs` folds each link segment's whole anchor (`ESC ] 8 ; ; URL ESC \` + visible text + `ESC ] 8 ; ; ESC \`) into the segment's *first* cell via `Buffer::cell_mut().set_symbol()`; the terminal's overflow paints the glyphs across the segment's columns. **Folding the full text is load-bearing**: ratatui 0.30 measures cell width as `symbol().width()`, so the URL bytes inflate the cell — the old prepend-open/append-close split dropped a character per link (`to_skip` reassigns every diff iteration, skipping exactly the cell after the inflated one; the fold's overflow covers it). Two residual costs of the inflated width: (a) a cursor on a folded link cell would invert the whole link / vanish on its 2nd char — avoided by `suppress_lines`, which leaves the cursor's/selection's own body lines un-anchored so the active line is always clean plain text; (b) `invalidated` re-renders extra cells per keystroke on link-dense mail (inefficient, usually invisible — the config gate exists to turn it off if it flickers). Single-cell segments are skipped (no interior cell to absorb the overflow). History: removed 2026-06-10 (commit 1103263) while chasing a cursor bug that turned out to be unrelated (cursor used coloured `REVERSED` → invisible on links, fixed 8540585; and over-width lines desynced rows, fixed 0cb53b8), then restored behind the toggle once those were fixed. `LinkSegment` tracking + the blockquote `> ` +2 shift also feed the `yl` yank-highlight.
- **Reader yanks — OSC 52 structural** — done. `Y` / `yp` / `yl` extract from the Block-IR (`mail/text.rs`) and pipe through `ui/clipboard.rs`. Default path: OSC 52 (`ESC ] 52 ; c ; <base64> ESC \`) to stdout, wrapped in `\x1bPtmux;…\x1b\\` (with inner `\x1b` doubled) under `$TMUX` and `\x1bP…\x1b\\` under `$STY` so the sequence escapes multiplexers. Payloads whose wrapped envelope exceeds `OSC52_MAX_BYTES` (100 KB) are rejected with a "set [reader].clipboard" hint, since terminals truncate silently above their own caps. Fallback: `[reader].clipboard = ["wl-copy", …]` spawns a `std::thread` worker that pipes stdin, mirroring `store/sync.rs` (`mpsc` + `AppEvent::Wake` on completion, drained by `App::poll_clipboard`); the fallback bypasses the size guard so large yanks just go through. Cursor is a body-relative `reader_cursor_line` clamped into the viewport each draw; `yp` resolves it through a new parallel `LaidOutBody.line_block_idx`, `yl` walks `LaidOutBody.links` for the first segment at/after the cursor (falls back to the first link in the body). `Screen::Inbox` was boxed (`parsed: Option<Box<ParsedBody>>`) to keep the enum under the `large_enum_variant` threshold.
- **Highlight on yank** — done. `yp` / `yl` and visual-mode `y` capture their cell ranges via `LaidOutBody::block_ranges` / `link_segment_ranges` / `selection_cell_ranges` and stash a `YankHighlight { ranges, expires_at }` on `InboxScreen` (`Y` whole-body is intentionally not flashed — too noisy, most of the body is offscreen). `reader::paint_yank_highlight` flips `Modifier::REVERSED` on covered cells each frame until expiry; `main::expire_yank_highlight` clears the state in `tick`. The main loop tightens `recv_timeout` to the remaining lifetime when a highlight is armed so the flash actually lasts the configured `[reader].yank_highlight_ms` (default 150 ms) instead of the 250 ms heartbeat. `0` disables. Highlight is reset alongside the reader cursor on message-body change and scope switch.
- **Visual mode in the reader** — done. `v` / `V` from Normal+Reader enter char-wise / line-wise visual (`Mode::Visual` + `InboxScreen.visual: Option<VisualState>` paired strictly via `App::enter_visual` / `exit_visual`). `hjkl`, `gg`, `G`, `0`, `$` extend; movement scrolls to follow (`InboxScreen::move_reader_cursor` + `follow_cursor`). `y` yanks the selection via the existing `clipboard::yank` sink and exits to Normal; `Esc` cancels; same-kind `v`/`V` toggles off, opposite-kind swaps via `set_visual_kind`. Selection text comes from `LaidOutBody.line_text` (per-line plain text derived from final layout spans — IR-via-layout, not cell-scraped); `extract_selection` cuts char-wise at endpoints, returns whole lines for line-wise. Selection cells render via `paint_selection`, which flips `Modifier::REVERSED` on covered cells (computed from `selection_cell_ranges` on `LaidOutBody`); the cursor cell is also REVERSED so the active end is visible even on empty rows. In `Mode::Visual` the reader draw skips the cursor-into-viewport clamp so the user can scroll selection past viewport edges; columns still get clamped to the line's char count once `line_text` is in hand. Cursor / visual state reset on message-body change and `switch_to_scope`.
- **Address completion (To / Cc / Bcc)** — done. Two sources feed the compose-tab popup, merged with external first then native, deduped by lowercase email. Native side (`mail/addressbook.rs`) walks each account's `[accounts.<name>].sent` folder once at startup via `start_addressbook_worker` — header parse goes through `mail::parse::parse_headers` (same lazy-header path `store::scan` uses), addresses harvested from `From / Reply-To / To / Cc / Bcc`, dedupe is by lowercased email and first non-empty display name wins. External side (`mail/addressbook_external.rs`) runs the configured `[compose.address_book].query_command` per query with the prefix as the trailing argv element; stdout parses mutt-protocol tab-separated `email[TAB name[TAB extra…]]` lines with the first line dropped as a status header (khard prints "searching for…"). Worker plumbing is the same `std::thread` + `mpsc` + `AppEvent::Wake` shape as `store::sync` / `store::compose::start_send_worker`. `App.poll_address_book` drains the startup walk into `App.address_book`, drains the in-flight external receiver into the active compose tab's popup, and dispatches the queued query when the debounce elapses (one in-flight worker at a time — newer queries replace the pending slot and fire after the current run reports). `App.refresh_address_complete` runs after every consumed compose-tab keystroke and derives `ComposeScreen.address_complete`: token = the trimmed segment between the previous `,` (or BOL) and the cursor; popup opens only when `token.len() >= [compose.address_book].min_chars` (default 2); inside `Name <…` the popup is suppressed so hand-edits of an already-accepted address don't get covered. Esc dismisses and parks the dismissed token on `ComposeScreen.address_complete_suppressed` so refresh doesn't immediately reopen on the same prefix; the next typed character clears the park (token diverges). Accept (Tab/Enter on a non-empty list) inserts `Name <email>` (or bare email when no name known) and appends `", "` iff the cursor was at end-of-field. Tab on an empty list falls through to the field-cycle so the form is still navigable when matches don't exist. Popup is rendered by `ui/address_complete.rs` mirroring `draw_from_picker`'s visual grammar (yellow border, `Clear`'d background, two-column rows: name | email); anchored on the focused recipient row's `Rect` stashed each draw in `ComposeScreen.last_complete_anchor`. Main loop's `recv_timeout` clamp now folds in `App.address_debounce_remaining()` alongside the yank-highlight deadline so the dispatcher fires within the configured `debounce_ms` (default 150) without busy polling. `[compose.address_book]` config keys: `query_command` (default unset → external source off), `debounce_ms` (default 150), `min_chars` (default 2). External worker failures (missing binary, non-zero exit) surface in `App.status_error`; popup keeps working on the native source.
- **Mouse-drag selection in the reader** — done. `[reader].mouse` (default `true`) gates `EnableMouseCapture`; `tty::enter` / `tty::leave` take the bool and the panic hook unconditionally emits `DisableMouseCapture` (harmless if never enabled). `Event::Mouse(_)` routes through `main::process_event` to `ui::mouse::handle`. `MouseDown(Left)` inside `InboxScreen.last_reader_inner` (set per-frame by `reader::draw`, cleared at the top of `app::draw` so a hidden pane never leaks a stale rect) anchors body-relative `(line, col)` on `InboxScreen.mouse_drag_anchor` and parks the reader cursor there. The first `Drag` promotes the gesture to `Mode::Visual` (char-wise) anchored at the press cell and updates the cursor (`follow_cursor` scrolls if needed); `Up(Left)` yanks via `keys::yank_visual` (now `pub(crate)`) and clears the anchor. A click (Down/Up at the same cell) leaves the cursor where it landed without entering visual or yanking — no `-- VISUAL --` flash. Wheel-over-reader nudges `reader_scroll` by 3 lines per notch; wheel outside the reader is ignored. Mouse events are suppressed in `Mode::Command` / `LinkPick` / `Search` so a stray click doesn't yank focus mid-input. `mouse_drag_anchor` resets alongside `reader_cursor_line/col` on body change and scope switch.

- **List-pane multi-select (visual-line)** — done. `v` / `V` from Normal + List focus anchor a vim-style visual-line selection over the message list (`App::toggle_list_visual`, gated on `Pane::List`); the anchor lives on `InboxScreen.list_visual: Option<usize>` and the other end follows `selected`, so the range is `min(anchor, selected)..=max(anchor, selected)`. There's no char/line distinction in a row list so both keys do the same thing; `j`/`k` extend, `v`/`V` again or `Esc` cancels. Action keys operate on the whole range with a single batched undo: `a` archive, `d` trash (per-row account routing — each message lands in its own account's role folder, rows on accounts without that role are skipped and counted), `m`/`*`/`x` toggle Seen/Flagged/Trashed (each row toggles independently). `:archive` / `:spam` / `:trash` / `:mv <folder>` are range-aware too — the selection survives into `Mode::Command` (vim's `:'<,'>`) so the cmdline acts on it; `exit_command` drops it on Esc-from-cmdline. The range members are read via `InboxScreen::row_at` (search-aware, like `selected_message_row`) so multi-select works in `/` search results as well as the threaded view. Range moves/flags collect `UndoAction`s and record one `UndoAction::Batch` (mirroring `D`); `move_msgid_to` / `toggle_flag_msgid` are the msgid-targeted primitives the loop drives. `cmdline::move_named` / `:mv` / `flag_selection` branch to the range path when `list_visual.is_some()`, else fall back to the single selected row. Rendering: `ui/list.rs::paint_list_visual` flips `Modifier::REVERSED` on every range row except the cursor row (which keeps the `List` widget's blue highlight), mirroring the reader's visual paint; the cmdline status row shows `-- VISUAL LINE -- N selected`. `list_visual` resets on scope switch, folder cycle, search entry/exit, `D`, and after any range action consumes it.

- **Reader cursor mode** — done. The reader pane has a real, independently-movable cursor (`InboxScreen.reader_cursor_line/col`) in Normal mode, not just Visual: `j`/`k` move it one body line (scroll follows via `follow_cursor`), `h`/`l`/`0`/`$` move within the line, `gg`/`G` jump to body top/bottom, `Ctrl-d`/`u`/`f`/`b` page the cursor (`App::page_move` → `InboxScreen::page_cursor`, half = `inner/2`, full = `inner-2` overlap). `Ctrl-e`/`Ctrl-y` (`App::scroll_reader_line`) scroll the viewport one line *without* moving the cursor — the draw-time keep-in-view clamp slides the cursor to the viewport edge, matching vim; the same clamp covers mouse-wheel scroll. The clamp no longer force-snaps the cursor to viewport-top (the old "passive shadow" behavior is gone). The cursor renders as a REVERSED cell in Normal mode so `v`/`V`/`yp`/`yl`/`gx`/`gs` targets are visible. The old direct-`reader_scroll` mutation for `j`/`k`/`G` and `InboxScreen::scroll_page` are removed.

- **Reader attachments — inline + cursor-addressable** — done. Attachments render as the first body lines (one `Attach: [n] filename (size)` chip row each, compose-style, continuation rows indented under the label, plus a blank separator), emitted by `reader::layout_with_images` before the block walk so the reader cursor can land on them and the image two-pass leaves their indices untouched. Tracked in `LaidOutBody.attachments: Vec<AttachmentSlot { line }>` (slot position = 0-based attachment index); `reader::draw` stashes their body lines on `InboxScreen.last_attachment_lines` each frame. The old pinned bottom strip (`render_attachment_strip`) is gone. `layout` / `layout_with_images` gained `attachments: &[Attachment]` + `attach_pick: Option<&str>` params (all callers updated, including the `yp`/`yl` re-layouts so cursor↔block indices stay aligned). Keys (Reader focus, under the `pending_g` latch): `gx` opens the attachment under the cursor in `[reader].browser`, `gs` saves it to cwd, `gd` drags it via `[reader].drag` — all via `InboxScreen::attachment_under_cursor` (exact chip-row match, else the sole attachment, else a "use gf" hint). `gf` enters `Mode::AttachmentPick` (digit buffer `App.attachment_pick_buf`, mirrors `LinkPick`): the `[n]` chip tags highlight yellow-on-black for matching candidates, `<Enter>` opens by number. `cmdline::{save,open,drag}_attachment_index` are the shared 1-based cores behind `:save` / `:open-attachment` / `:drag` / the cursor keys / the picker. Mouse events suppressed in `AttachmentPick` like the other capture modes.

- **Vim word motions + visual-block, both panes** — done. A shared word-boundary scanner (`ui/words.rs::word_motion` over `&[String]`, char-index coords) backs `w`/`W`/`b`/`B`/`e`/`E` in the reader (`InboxScreen::reader_word` reads the per-frame `last_reader_body_line_text`, wired into both the Normal keymap and the `MotionTarget` impl) and the composer (`BodyEditor::word_move` over `textarea.lines()`; all six word motions now route through the scanner so the panes can't drift). `motion.rs` gained `WordForwardBig`/`WordBackBig`/`WordEndBig` + `W`/`B`/`E` key mapping. Reader `Y`/`yy` now yank the current line (vim line-wise; `yank_line` reads `last_reader_body_line_text[cursor]` and flashes the row); whole-body yank moved to `yie`/`yae` (vim-textobj-entire) via the existing `yi`/`ya` pending-`y` prefix collector; composer `Y` aliases `yy`. Reader `G` lands at column 0 of the last line (was `u16::MAX`/EOL), matching `gg`. Visual-block (`Ctrl-V`, arrives as `Char('v')`+CONTROL): a third `VisualKind::Block` on both the reader and composer enums, normalizing rows/cols *independently* (rectangle) unlike the lexicographic Char/Line. Reader is read-only → select + yank (block branches in `reader.rs::extract_selection` / `selection_cell_ranges`, Ctrl-V entry in the `inbox_normal` Ctrl block guarded on Reader focus, Ctrl-V swap in `visual()` before the plain-`v` arm). - **Vim operator-pending engine (composer) + reader read-only motions** — done. The composer's toy `Pending` latch (only `dd`/`yy`/`gg`) is replaced by a real operator × motion × text-object × count engine in `ui/compose_body.rs` (`OpState`/`Operator`/`FindKind`). The keystone is a *pure* resolver path in `ui/motion.rs`: `resolve_motion_span(lines, cursor, motion, count) -> MotionSpan` computes endpoints without moving the cursor (so it unit-tests with no textarea), and `span_to_region` normalises a span into a `Region` (inclusive→exclusive +1 at EOL, linewise collapse). `BodyEditor::apply_operator` is the single sink for both Normal and Visual operators. Counts multiply (`2d3w`). Vim quirks handled: `cw`→`ce` on non-blank, `dw`-at-EOL clamp, `f`/`t` inclusive vs `F`/`T` exclusive. Discrete keys: `r{char}` (count-aware), `R` (new `BodyMode::Replace` overtype), `~`, `J`/`gJ`, `s S C D`. Visual mode routes `> < gu gU g~ gw gq` through `apply_operator` and accepts text objects (`viwd` == `diw`). `BodyEditor::handle_key` now takes `text_width` (threaded from `cfg.compose.text_width` at the `compose.rs` callsite). Reader (read-only, `ui/keys.rs` + `ui/app.rs`): counts (`3j`) via `App.pending_count`, `{`/`}` paragraph motions (`InboxScreen::reader_paragraph`), `H`/`M`/`L` viewport positions, `zz`/`zt`/`zb` scroll-positioning via `App.pending_z` + `reader_scroll_cursor`, `ge`/`gE`, and operator-yanks (`yw`/`yiw`/`y}`/`yi"`…) that reuse `resolve_motion_span`/`resolve_text_object` over `last_reader_body_line_text` and pipe through the existing `dispatch_yank` sink. `f` stays the Vimium link-picker (no vim find-char conflict). v1 drift: case/reflow/indent operators are 2 undo steps (delete+insert); dot-repeat (`.`) and named registers deferred.

## Commands

All commands go through `task` (go-task); `Taskfile.yml` is the canonical surface. `task --list` enumerates. Highlights:

- `task dev` — main loop. `bacon` auto-rebuilds and restarts against `dev/config.toml`.
- `task run` — one-shot run against the dev config.
- `task ci` — `fmt:check + lint + test`, what CI would run.
- `task doctor` — smoke-check the devshell and fixtures.
- `task snap:review` / `task snap:accept` — `cargo insta` for renderer golden snapshots (golden cell buffers per HTML fixture).

Prefer the tasks over raw `cargo run`: the wrappers pass `--config dev/config.toml` so the app uses the in-repo fixture maildir and the stub msmtp instead of real mail. Single-test invocation is plain `cargo test <name>`.

## Constraints that catch agents

These are easy to flagrantly violate. The full set lives in `DESIGN.md`; this is a triage list to scan before editing:

- **No `async`/`await`, no tokio.** Slow work (mbsync trigger, msmtp send, maildir scans, search, large-batch threading, the browser-fallback spawn for `:open`) runs on `std::thread` workers; results flow back through `mpsc` channels. The main loop is event-driven, not timer-driven: a dedicated input thread and every worker push into one unified `AppEvent` channel (`src/ui/events.rs`), and `main.rs` blocks on `recv_timeout` so it wakes the instant any source has something to say. The timeout is a safety-belt fallback, not the primary trigger. Adding tokio is a redesign — raise it explicitly before doing it.
- **No GUI, no webview, no JavaScript anywhere.** The pivot to TUI means `eframe` / `egui` / `wgpu` / `vello` / `blitz-*` are out. The HTML body is parsed with `html5ever` and rendered into a ratatui cell buffer. The "no HTML engine that can execute code" property is a depended-on security guarantee — `html5ever` is a parser, not an engine.
- **Remote content is never fetched.** The HTML walker resolves `cid:` against MIME parts and decodes `data:` inline; `http(s)://` images render as `[remote image: alt]` placeholders. There is no per-message "load images" toggle. The escape hatch is `:open`, which writes the message HTML to a tempfile (with `cid:` rewrites) and hands it to the user's browser via the configured command.
- **The app never writes its TOML config.** Runtime `:set` is session-only. Persistent state belongs in `$XDG_CACHE_HOME/epost/` (the sqlite index), never in config.
- **Keys on `Message-ID`, never on file path.** mbsync rewrites maildir paths and the info-flag suffix on every sync, so paths go stale constantly.
- **Approximate-fidelity HTML by design.** Cell-grid rendering of CSS will not be pixel-faithful; that is a feature, not a bug. Don't add CSS-engine deps to "improve fidelity." If a class of mail looks too broken, the answer is `:open` and the system browser — not a webview.

## Dev environment / fixtures

`devenv.nix` provides the devshell (Rust toolchain, `mold` linker, `bacon`, `mbsync`, `msmtp`, `cargo-insta`). `direnv allow` activates it.

`dev/` is the in-repo dev harness:

- `dev/config.toml` — the TOML the binary reads under `task dev` / `task run`.
- `dev/maildir/` — in-tree synthetic maildir for the `dev` account (maildir++ layout). `dev/maildir/README.md` lists the cases covered; outstanding fixture gaps are tracked in `TODO.md`.
- `dev/maildir-work/` — second-account fixture (also maildir++). Mirrored into `dev/scratch/maildir-work/` on first run.
- `dev/maildir-verbatim/` — third-account fixture exercising the verbatim / nested layout (real subdirectories, `Archive/2024/` nested two deep). Backs `[accounts.verbatim]` in `dev/config.toml`; mirrored into `dev/scratch/maildir-verbatim/` on first run so both code paths stay live under `task dev`.
- `dev/msmtp-stub` — shell script wired into `[smtp].command` in dev; captures outbound mail to `/tmp/epost-sent/` instead of sending.
- `dev/fixtures/` — hand-crafted HTML and image fixtures for renderer tests. Step 1's `welcome.html` + `welcome.png` from the GUI era can be reused as the first cell-rendering target.
